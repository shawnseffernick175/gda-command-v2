 S-009 Opportunity Detail & OODA Spec

## Purpose

This specification defines the Opportunity Detail experience introduced in Sprint S-009. It establishes the API contract, data shape, UI layout, and OODA analysis structure that engineering will implement in the following sprint. This spec governs the clickthrough destination when a user selects any opportunity from Ops Tracker or Pipeline.

All existing S-007 and S-008 behavior — qualify safety gates, pipeline read-only view, audit log, smoke tests, and response envelope contracts — remains unchanged. This spec adds a new surface; it does not modify any existing surface.

---

## Non-negotiable Requirements

1. Every opportunity row in Ops Tracker and every opportunity row in Pipeline must be clickable and must navigate to this detail view.
2. The detail view must present real analysis, not raw field dumps. At minimum: core fields, an executive summary, and clearly labeled analysis blocks.
3. The OODA block is mandatory. It must appear on every opportunity detail view. It must structurally separate source-backed facts (Observe, Orient) from AI-assisted interpretation and recommended action (Decide, Act).
4. Every claim in the analysis that draws on external information must be traceable to a source object in `data.sources`. No invented citations. No inline URLs without a corresponding source record.
5. If no sources are available, the sources array must be present and empty. The UI must render an explicit "No external sources available" state rather than hiding the section.
6. The API response must follow the standard GDA envelope: `success`, `workflow`, `action`, `dryRun`, `data`, `meta`, `error`.
7. The endpoint is read-only. No write operations are permitted in this sprint.
8. Future learning fields must be reserved in the response shape even if unpopulated in the first implementation. They must not be omitted from the contract.
9. Layout and spacing are product requirements. The detail view must have clear section hierarchy, readable typography, and visible section labels. A flat wall of fields is not acceptable.
10. The persistent financial KPI strip must remain visible above this view. The local summary strip must reflect Ops Tracker or Pipeline context depending on entry point.

## API Contract

### Endpoint
GET /api/opportunities/:id/detail

### Request

| Parameter | Location | Type   | Required | Description                        |
|-----------|----------|--------|----------|------------------------------------|
| id        | path     | string | yes      | Opportunity UUID from Postgres      |
| dryRun    | query    | bool   | no       | If true, return shape with no DB hit (for smoke testing) |

### Response Envelope

```json
{
  "success": true,
  "workflow": "gda-opportunity-detail",
  "action": "read",
  "dryRun": false,
  "data": { ... },
  "meta": {
    "requestedAt": "ISO8601",
    "respondedAt": "ISO8601",
    "opportunityId": "uuid",
    "sourceCount": 3,
    "analysisGeneratedAt": "ISO8601 or null",
    "coverageFlags": {
      "hasOoda": true,
      "hasSources": true,
      "hasLearning": false
    }
  },
  "error": null
}
```

On failure:

```json
{
  "success": false,
  "workflow": "gda-opportunity-detail",
  "action": "read",
  "dryRun": false,
  "data": null,
  "meta": {
    "requestedAt": "ISO8601",
    "respondedAt": "ISO8601",
    "opportunityId": "uuid"
  },
  "error": {
    "code": "OPPORTUNITY_NOT_FOUND",
    "message": "No opportunity found for the given ID.",
    "detail": null
  }
}
```

### `data` Shape

```json
{
  "opportunity": { ... },
  "analysis": { ... },
  "ooda": { ... },
  "sources": [ ... ],
  "learning": { ... }
}
```

### `data.ooda` Structure

The OODA block is mandatory. Each of the four sub-objects must be present. If a section has no content yet, it must be present with an empty `items` array and a null `summary`.

```json
{
  "observe": {
    "summary": "string | null",
    "items": [
      {
        "label": "string",
        "value": "string",
        "source_ids": ["source-uuid-1"]
      }
    ]
  },
  "orient": {
    "summary": "string | null",
    "items": [
      {
        "label": "string",
        "value": "string",
        "source_ids": ["source-uuid-1"],
        "type": "fact | inference | risk | strength"
      }
    ]
  },
  "decide": {
    "summary": "string | null",
    "options": [
      {
        "label": "string",
        "rationale": "string",
        "recommended": true
      }
    ]
  },
  "act": {
    "summary": "string | null",
    "next_steps": [
      {
        "action": "string",
        "owner": "string | null",
        "due_date": "ISO8601 date | null",
        "priority": "high | medium | low"
      }
    ]
  }
}
```

**OODA intent:**
- `observe`: Source-backed facts only. What do we know from documents, databases, published data. Every item must reference at least one source_id from `data.sources` if it comes from external information.
- `orient`: Interpretation of the observed facts in context of Envision / GDA OU3 capabilities, past performance, incumbent situation, competitive position. Items may be facts or labeled inferences.
- `decide`: Decision options with explicit rationale. The recommended option must be marked `recommended: true`.
- `act`: Concrete next steps with owner placeholders and priority.

### `data.sources` — Normalized Source Objects

The sources array contains every external reference used in the analysis or OODA block. If no sources exist, the array must be present and empty (`[]`).

```json
[
  {
    "id": "string (UUID)",
    "title": "string",
    "type": "sam_gov | beta_sam | news | report | contract_award | incumbent_profile | agency_forecast | other",
    "url": "string | null",
    "publisher": "string | null",
    "published_at": "ISO8601 date | null",
    "retrieved_at": "ISO8601 timestamp | null",
    "snippet": "string | null",
    "relevance_reason": "string"
  }
]
```

| Field            | Description                                                                 |
|------------------|-----------------------------------------------------------------------------|
| id               | UUID used by OODA items to reference this source via `source_ids`           |
| title            | Human-readable title of the source document or article                      |
| type             | Controlled vocabulary for source type                                       |
| url              | Inspectable URL (must be real; do not invent)                               |
| publisher        | Publishing organization                                                     |
| published_at     | Publication date if known                                                   |
| retrieved_at     | When this source was fetched/indexed                                        |
| snippet          | Brief relevant excerpt or description — not a full reproduction             |
| relevance_reason | Plain English: why this source is cited for this opportunity                |

### `data.learning` Fields

Reserved for future analyst-feedback and model-improvement infrastructure. All fields must be present in the response; values will be null in the first implementation.

| Field              | Type           | Description                                                        |
|--------------------|----------------|--------------------------------------------------------------------|
| learning_notes     | string \| null | Operator or analyst notes for improving future analysis            |
| feedback_submitted | bool           | Whether feedback has been submitted for this analysis              |
| feedback_at        | string \| null | ISO8601 timestamp of last feedback submission                      |
| source_count       | number         | Count of sources used in current analysis                          |
| coverage_gaps      | string[]       | Areas where source coverage was weak or absent                     |
| next_review_at     | string \| null | Suggested date for re-analysis                                     |

## UI Layout

The Opportunity Detail view renders below the persistent financial KPI strip and the tool-specific local summary strip. It does not replace or suppress those global elements.

### Section Order (top to bottom)
[ Persistent Financial KPI Strip ]
[ Local Summary Strip — Ops Tracker or Pipeline context ]
─────────────────────────────────────────────────────────
[ Breadcrumb: Ops Tracker > Opportunity Title ]
[ Page Header: Opportunity Title + Status Badge + Score ]
─────────────────────────────────────────────────────────
[ SECTION 1: Core Fields ]
Two-column label/value grid:
Agency | Department | Solicitation Number | NAICS | PSC
Estimated Value | Probability of Win | Due Date
Set-Aside | Place of Performance | Incumbent | Tags
[ SECTION 2: Executive Summary ]
Plain paragraph. Source attribution if applicable.
[ SECTION 3: OODA Analysis ]
Sub-section header: "Observe — What We Know"
Labeled fact items with source badges
Sub-section header: "Orient — What It Means"
Labeled inference/risk/strength items, type badges, source badges
Sub-section header: "Decide — Options & Recommendation"
Option cards; recommended option visually distinguished
Sub-section header: "Act — Next Steps"
Prioritized action list with owner and due date fields
[ SECTION 4: Strengths / Risks ]
Two-column layout: Strengths column | Risks column
Plain bulleted lists, no charts
[ SECTION 5: Competitive Landscape ]
Plain text. Visible only if data is present; otherwise shows
"No competitive intelligence available for this opportunity."
[ SECTION 6: Sources ]
Labeled section header: "Sources"
Table or card list: Title | Type | Publisher | Published | Snippet | Relevance
If empty: "No external sources are associated with this analysis."
[ SECTION 7: Learning & Feedback ]
Read-only in first implementation.
Shows: source count, coverage gaps (if any), last analyzed timestamp.
Feedback controls: reserved, rendered as disabled placeholder.

### Typography and Hierarchy Rules

- Section headers: bold, larger than body, with a visible top border or spacing separator.
- Sub-section headers within OODA: medium weight, visually subordinate to section headers.
- Labels in core fields grid: muted/secondary color, uppercase small caps or bold small text.
- Values: normal weight, full contrast.
- Status badge: color-coded pill. `discovery` = gray, `qualified` = blue, `pipeline` = green, `lost` = red, `won` = gold.
- Score: numeric display with a label, no bar chart in first implementation.
- Source badges on OODA items: small clickable chip that scrolls to the source in Section 6.

## States & Empty Cases

| State                         | Behavior                                                                 |
|-------------------------------|--------------------------------------------------------------------------|
| Opportunity not found         | Show error state with message from `error.message`. No blank page.       |
| Analysis not yet generated    | Show core fields. Analysis sections show "Analysis pending." placeholder. |
| OODA sections empty           | Each sub-section renders its header and an explicit empty message.        |
| Sources array empty           | Sources section renders with "No external sources available."             |
| Loading                       | Skeleton placeholder for each section block. No spinner-only states.     |
| Network / API failure         | Show inline error: workflow name, action, error code, message. No silent blank. |
| dryRun=true response          | Render a visible "DRY RUN — No live data" banner at top of view.         |

## Future Learning Hooks

The following fields and infrastructure points are reserved in this spec for future implementation. Engineering must not omit them from the data contract even if values are null.

| Hook                        | Location               | Purpose                                                              |
|-----------------------------|------------------------|----------------------------------------------------------------------|
| `data.learning.*`           | API response           | Feedback, coverage gaps, re-analysis scheduling                      |
| `data.analysis.confidence`  | API response           | Model-assigned confidence score for the analysis                     |
| `data.analysis.analyst_feedback` | API response      | Operator override or correction of AI analysis                       |
| `data.analysis.analysis_version` | API response      | Tracks which prompt/model version produced the analysis              |
| `data.ooda.observe[].source_ids` | OODA items        | Enables citation linking and source audit                            |
| `data.ooda.orient[].type`   | OODA items             | Enables filtering facts from inferences in future analysis views     |
| Feedback UI controls        | Section 7              | Disabled in first implementation; wired in a future sprint           |
| `next_review_at`            | `data.learning`        | Future: scheduled re-analysis trigger                                |
| `coverage_gaps`             | `data.learning`        | Future: surface weak intelligence areas to operator                  |
| `source.retrieved_at`       | Source objects         | Future: staleness detection and re-fetch scheduling                  |
